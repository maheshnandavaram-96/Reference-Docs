JavaScript Type Coercion

Coercion is JavaScript's automatic (or manual) conversion of values from one type to another. It's one of the most misunderstood parts of the language — and one of the most powerful once you understand the rules.

What happens here?

// Each of these "just works" — but WHY?
1 + "2"         // → "12"   (number coerced to string)
"5" - 3         // → 2      (string coerced to number)
!!""            // → false  (string coerced to boolean)
!!"hello"       // → true
[] == 0          // → true   (array → 0!)
null == undefined // → true   (special rule)

All of this is governed by a small set of abstract operations defined in the ECMAScript spec. Once you know them, nothing surprises you.

Implicit coercion
JavaScript converts types automatically behind the scenes. E.g., using + with a string and number.
Explicit coercion
You manually convert types using built-ins like Number(), String(), Boolean().

JavaScript's 8 Types

Everything in JavaScript is one of these types. Coercion is the engine that converts between them.

The primitive types are: number, string, boolean, undefined, null, symbol, bigint. Everything else is an object.

Implicit vs Explicit Coercion

JavaScript triggers coercion automatically in many contexts — arithmetic, comparison, template literals, logical operators, etc.

// Arithmetic context
"5" - 1        // → 4     (string "5" → number 5, then subtract)
"5" * 2        // → 10    (same)
"5" + 1        // → "51"  (DIFFERENT! + triggers string concatenation)

// Boolean context (if/while conditions)
if ("") {}       // falsy — never runs
if ("hello") {}  // truthy — runs
if (0) {}        // falsy — never runs
if ([]) {}       // truthy! empty array is truthy

// Template literals (always to string)
`Value: ${42}`   // → "Value: 42"
`Is: ${true}`   // → "Is: true"

You control explicit coercion explicitly using built-in functions. This is always preferred for clarity.

// To Number
Number("42")     // → 42
Number("")       // → 0
Number(true)     // → 1
Number(null)     // → 0
Number(undefined) // → NaN
parseInt("42px") // → 42  (stops at non-numeric)

// To String
String(42)       // → "42"
String(true)     // → "true"
String(null)     // → "null"
(42).toString()  // → "42"

// To Boolean
Boolean(0)       // → false
Boolean("")      // → false
Boolean({})      // → true
Boolean([])      // → true  (!!)
!!"hello"        // → true  (double-bang trick)

Abstract Operation: ToPrimitive

Abstract Op

When JavaScript needs a primitive value from an object, it calls the internal ToPrimitive operation. This is the gateway — every other abstract op (ToNumber, ToString) calls ToPrimitive first when it receives an object.

Signature
ToPrimitive(input, preferredType?) — preferredType is either "number" (default) or "string"

How it works — step by step

Object received
→
Check Symbol.toPrimitive
→
If missing: try hint methods
→
Return primitive
hint: "number" Default. Tries valueOf() first, then toString() ▼

When hint is "number" (the default), JS tries:

  1. Call obj[Symbol.toPrimitive]("number") if it exists
  2. Else call obj.valueOf() — if it returns a primitive, use it
  3. Else call obj.toString() — if it returns a primitive, use it
  4. Else throw TypeError
const obj = {
  valueOf() { return 42; },       // returns a primitive
  toString() { return "hello"; }
};

obj + 1   // hint: "number" → valueOf() → 42 → 42 + 1 = 43
+obj     // hint: "number" → valueOf() → 42
hint: "string" Tries toString() first, then valueOf() ▼

When hint is "string" (e.g., template literals, String()), the order reverses:

  1. Call obj[Symbol.toPrimitive]("string") if it exists
  2. Else call obj.toString() — if it returns a primitive, use it
  3. Else call obj.valueOf()
const obj = {
  valueOf()  { return 42; },
  toString() { return "hello"; }
};

String(obj)         // hint: "string" → toString() → "hello"
`${obj}`            // hint: "string" → toString() → "hello"
obj.toString()      // → "hello"
Symbol.toPrimitive Override the entire conversion with a single method ▼

You can define [Symbol.toPrimitive] to take full control. It receives the hint as an argument.

const smartObj = {
  [Symbol.toPrimitive](hint) {
    if (hint === "number")  return 100;
    if (hint === "string")  return "one hundred";
    return true;  // hint === "default"
  }
};

+smartObj         // → 100    (hint: "number")
`${smartObj}`     // → "one hundred" (hint: "string")
smartObj + ""    // → "true"  (hint: "default")
When does "default" hint occur?
When using +, ==, or the >/< operators without a clear context. The spec says + on an object uses "default" hint for ToPrimitive.

Built-in objects and ToPrimitive

// Arrays: toString() joins elements with comma
[1, 2, 3].valueOf()   // → [1,2,3]  (not a primitive!)
[1, 2, 3].toString()  // → "1,2,3"
[1, 2, 3] + ""        // → "1,2,3"
[1] + [2]             // → "12"  !

// Dates: prefer string hint over number
new Date()[Symbol.toPrimitive]("string")  // → "Thu Apr 10 2025..."
new Date()[Symbol.toPrimitive]("number")  // → 1744243200000 (timestamp)
new Date() + ""                           // → date string (Date uses "default" → string)

// Plain object
({}).valueOf()   // → {} (the object itself, not a primitive)
({}).toString()  // → "[object Object]"
{} + ""          // → "[object Object]"

Abstract Operation: ToBoolean

Abstract Op

Called in boolean contexts: if/while conditions, &&/||/! operators, ternary ? :, and Boolean() explicitly.

The complete falsy list (only 8 values)
false, 0, -0, 0n (BigInt zero), "" (empty string), null, undefined, NaN — everything else is truthy.

Falsy values — exhaustive list

Boolean(false)→false
Boolean(0)→false
Boolean(-0)→false
Boolean(0n)→false
Boolean("")→false
Boolean(null)→false
Boolean(undefined)→false
Boolean(NaN)→false

Truthy surprises 😮

Boolean([])→true empty array is truthy!
Boolean({})→true empty object is truthy!
Boolean("0")→true non-empty string, even "0"!
Boolean("false")→true non-empty string!
Boolean(new Boolean(false))→true it's an object!
Boolean(Infinity)→true not in the falsy list

ToBoolean does NOT call ToPrimitive

Unlike ToNumber and ToString, ToBoolean is a simple lookup — it doesn't convert objects to primitives first. An object is always truthy, no matter what its valueOf() or toString() returns.

const sneaky = {
  valueOf() { return 0; },       // zero (falsy number)
  toString() { return ""; }      // empty string (falsy)
};

if (sneaky) console.log("runs!"); // → "runs!" (it's an object!)
Boolean(sneaky)                    // → true
Number(sneaky)                     // → 0 (uses valueOf)

// The common bug:
const val = 0;
if (val) { /* never runs */ }           // 0 is falsy
if (val !== undefined) { /* runs */ }  // 0 IS defined, use explicit check

Abstract Op

The reverse direction — wrapping a primitive in its object wrapper. This is what happens when you call a method on a primitive like "hello".toUpperCase().

Auto-boxing: the invisible ToObject

Every time you access a property or method on a primitive, JavaScript silently calls ToObject behind the scenes, then discards the wrapper:

// What you write:
"hello".toUpperCase()    // → "HELLO"
"hello".length           // → 5
(42).toFixed(2)          // → "42.00"
true.toString()           // → "true"

// What JS actually does (conceptually):
let temp = new String("hello");  // ToObject("hello")
temp.toUpperCase();                  // → "HELLO"
// temp is discarded immediately — you can't set properties on primitives!

// This is why this silently fails:
"hello".custom = 42;   // wrapper created, property set, wrapper discarded
"hello".custom;         // → undefined (new wrapper, property gone)
Don't use new Boolean() / new Number() / new String()
Wrapping primitives creates objects, which are always truthy regardless of their value. new Boolean(false) is truthy! Use Boolean(), Number(), String() without new.

The + Operator

The + operator is the most complicated coercion site in JavaScript. It can mean numeric addition or string concatenation — and the decision is made at runtime based on the types.

Algorithm How + decides what to do ▼
  1. Call ToPrimitive(left, "default") and ToPrimitive(right, "default")
  2. If either result is a string → concatenate both as strings (both go through ToString)
  3. Otherwise → add both as numbers (both go through ToNumber)
left + right
→
ToPrimitive each
→
Either a string?
→
Yes: concatenate
→
No: add numbers
// String wins — any string operand → concatenation
1 + "2"      // → "12"   (1 → "1", then "1"+"2")
"1" + 2      // → "12"   (2 → "2", then "1"+"2")
"1" + "2"    // → "12"   (both strings, concat)

// Both non-string primitives → numeric addition
1 + 2        // → 3
1 + true     // → 2    (true → 1)
1 + null     // → 1    (null → 0)
1 + undefined // → NaN  (undefined → NaN)
true + true  // → 2
null + null  // → 0

// Objects → ToPrimitive first
[] + []       // → ""    ([] → "", "" + "" = "")
[] + {}       // → "[object Object]"  ([] → "", {} → "[object Object]")
{} + []       // → 0     (as a statement, {} is parsed as a block!)
({}) + []     // → "[object Object]"

// Classic interview question:
1 + 2 + "3"  // → "33"   (left to right: 1+2=3, then 3+"3"="33")
"1" + 2 + 3  // → "123"  ("1"+2="12", then "12"+3="123")

Relational Operators: < > <= >=

These operators use an Abstract Relational Comparison algorithm — which also involves ToPrimitive and ToNumber.

Algorithm: a < b Abstract relational comparison ▼
  1. Compute px = ToPrimitive(a, "number")
  2. Compute py = ToPrimitive(b, "number")
  3. If both px and py are strings → compare lexicographically
  4. Otherwise → compare numerically (ToNumber both)
// Both strings → lexicographic (character code comparison)
"abc" < "abd"    // → true  (c < d)
"10" < "9"      // → true!  "1" < "9" lexicographically
"B" < "a"       // → true   uppercase letters < lowercase (ASCII)

// One or both non-strings → numeric
"5" > 4         // → true  ("5" → 5, 5 > 4)
null > 0        // → false (null → 0, 0 > 0 is false)
null == 0       // → false (null only == undefined)
null >= 0       // → true!  (null → 0, 0 >= 0)
null < 1        // → true   (null → 0, 0 < 1)

// NaN comparisons are always false
NaN < 1          // → false
NaN > 1          // → false
NaN == NaN        // → false  (NaN is not equal to itself!)
isNaN(NaN)        // → true   (use this to check)
⚠ The null >= 0 paradox

null > 0 is false, null == 0 is false, yet null >= 0 is true. How?

>= is computed as !(null < 0). Since null < 0 is false, !(false) is true. The == operator uses a completely different algorithm that special-cases null.

No coercion. Different types → always false. Same type → compare values directly. Use this by default.

Top Coercion Gotchas

These are the most common bugs caused by coercion — know them, and you'll write much safer JS.

1. typeof NaN === "number"
typeof NaN         // → "number"  (Not-a-Number IS a Number type!)
isNaN("hello")     // → true  (isNaN coerces first: Number("hello") = NaN)
Number.isNaN("hello") // → false (safer! doesn't coerce)
Number.isNaN(NaN)  // → true  ✓ use this
2. parseInt vs Number for strings
parseInt("42abc")  // → 42   (stops at non-numeric char)
Number("42abc")   // → NaN  (entire string must be numeric)
parseInt("0x1F") // → 31   (hex!)
parseInt("010")   // → 10   (not octal in modern JS — always use radix)
parseInt("010", 8)// → 8    (explicit octal)
parseInt("010", 10)// → 10  (explicit decimal — always do this)
3. Loose equality with arrays
[] == false    // → true  ([] → "" → 0, false → 0)
[] == 0       // → true  ([] → "" → 0)
[] == ""      // → true  ([] → "")
[] == ![]     // → true  (![] = false, then [] == false = true)
[0] == false  // → true  ([0] → "0" → 0, false → 0)
4. +[] and +{} weirdness
+[]    // → 0   (unary + → ToNumber → ToPrimitive([]) = "" → 0)
+{}    // → NaN (ToPrimitive({}) = "[object Object]" → NaN)
+""    // → 0   (ToNumber("") = 0)
+"3"   // → 3   (ToNumber("3") = 3)
+null  // → 0   (ToNumber(null) = 0)
+undefined // → NaN
5. The sort() trap
[10, 9, 2, 1, 100].sort()
// → [1, 10, 100, 2, 9]  (lexicographic! "10" < "2")

[10, 9, 2, 1, 100].sort((a, b) => a - b)
// → [1, 2, 9, 10, 100]  (numeric comparator — correct)
Summary — golden rules
  1. Use === instead of == (except for x == null)
  2. Use Number.isNaN() not isNaN()
  3. Use explicit coercion — Number(x), String(x), Boolean(x)
  4. Always pass a radix to parseInt()
  5. Always pass a comparator to .sort() for numbers
  6. Check for null/undefined explicitly before arithmetic
