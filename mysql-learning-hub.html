MON Introduction, Installation & First Connection Basics ▶

MySQL is a relational database management system (RDBMS). Data is stored in tables with rows and columns, and tables can relate to each other through keys. SQL (Structured Query Language) is the language you use to interact with it.

ℹ️Key Concepts: Database, Table, Row (Record), Column (Field), Schema, Primary Key, SQL vs MySQL
RDBMS — organizes data into structured tables with relationships between them
Schema — the blueprint of your database (table structure, data types, constraints)
Primary Key — a column that uniquely identifies each row
SQL categories — DDL, DML, DQL, DCL, TCL (you'll learn all of these)

TUE CREATE DATABASE & CREATE TABLE — Data Types DDL ▶

DDL (Data Definition Language) is used to define and modify the structure of your database. CREATE, ALTER, and DROP are the core DDL commands. Choosing the right data type is critical — it affects storage, performance, and data integrity.

💡Always specify a character set when creating databases. utf8mb4 supports emojis and all Unicode characters — use it by default.

WED Constraints — Rules That Protect Your Data DDL ▶

Constraints enforce rules on data at the database level — they prevent bad data from ever entering your tables, regardless of the application code. Learn to rely on the database itself as the last line of defense.

PRIMARY KEY
Uniquely identifies each row. Cannot be NULL. Usually an auto-incrementing integer.
NOT NULL
The column must always have a value. Prevents empty records for required fields like name or email.
UNIQUE
No two rows can have the same value in this column. Unlike PRIMARY KEY, a column can be UNIQUE and also be NULL.
DEFAULT
Value used if none is supplied during INSERT. Saves you from specifying common values every time.
AUTO_INCREMENT
MySQL automatically assigns the next integer value. Used almost universally for primary keys.
CHECK
Validates that a column value meets a condition. Example: age must be >= 0. Available natively in MySQL 8.0+.
sql — DDL All major constraints in one table 
CREATE TABLE employees (
  -- PRIMARY KEY + AUTO_INCREMENT
  emp_id       INT              AUTO_INCREMENT PRIMARY KEY,

  -- NOT NULL: required fields
  first_name   VARCHAR(50)     NOT NULL,
  last_name    VARCHAR(50)     NOT NULL,

  -- UNIQUE: no duplicates allowed
  email        VARCHAR(120)    NOT NULL UNIQUE,
  emp_code     CHAR(6)         UNIQUE,

  -- DEFAULT: automatic value
  department   VARCHAR(50)     DEFAULT 'General',
  is_active    BOOLEAN          DEFAULT TRUE,
  hire_date    DATE             DEFAULT (CURDATE()),
  created_at   TIMESTAMP        DEFAULT CURRENT_TIMESTAMP,

  -- Named composite UNIQUE constraint
  CONSTRAINT uq_emp_name UNIQUE (first_name, last_name, department)
);
sql — DDL CHECK constraints (MySQL 8.0+) 
CREATE TABLE products (
  product_id  INT              AUTO_INCREMENT PRIMARY KEY,
  name        VARCHAR(100)    NOT NULL,
  price       DECIMAL(10,2)  NOT NULL,
  stock       INT              DEFAULT 0,
  discount    DECIMAL(5,2)   DEFAULT 0,

  -- Inline CHECK: price must be positive
  CONSTRAINT chk_price   CHECK (price > 0),

  -- stock cannot be negative
  CONSTRAINT chk_stock   CHECK (stock >= 0),

  -- discount is a percentage 0–100
  CONSTRAINT chk_disc    CHECK (discount BETWEEN 0 AND 100),

  -- discount cannot exceed price
  CONSTRAINT chk_disc2   CHECK (discount < price)
);

-- This INSERT will FAIL — price is negative
INSERT INTO products (name, price) VALUES ('Widget', -5.00);
-- ERROR 3819 (HY000): Check constraint 'chk_price' is violated
⚠️CHECK constraints are enforced in MySQL 8.0.16+. In older versions, they are parsed but silently ignored. Always check your MySQL version with SELECT VERSION();
sql — DDL Add constraints to existing tables 
-- Add a UNIQUE constraint after table creation
ALTER TABLE employees
  ADD CONSTRAINT uq_email UNIQUE (email);

-- Add a NOT NULL constraint (requires MODIFY)
ALTER TABLE employees
  MODIFY COLUMN first_name VARCHAR(50) NOT NULL;

-- Add a CHECK constraint
ALTER TABLE employees
  ADD CONSTRAINT chk_salary CHECK (salary > 0);

-- Remove a named constraint
ALTER TABLE employees
  DROP CONSTRAINT chk_salary;

-- Show all constraints on a table
SELECT constraint_name, constraint_type
FROM information_schema.table_constraints
WHERE table_name = 'employees';
THU ALTER TABLE & DROP — Modifying Structure DDL ▶

Once tables are created, you'll need to modify them as your application evolves. ALTER TABLE lets you add, modify, or remove columns and constraints without losing data.

⚠️Production Warning: ALTER TABLE on large tables can lock the entire table for minutes. Use tools like pt-online-schema-change or MySQL 8's instant DDL in production.
sql — DDL All ALTER TABLE operations 
-- ADD a new column
ALTER TABLE customers
  ADD COLUMN loyalty_points INT DEFAULT 0;

-- ADD column at a specific position
ALTER TABLE customers
  ADD COLUMN middle_name VARCHAR(50) AFTER first_name;

-- MODIFY: change data type or constraints
ALTER TABLE customers
  MODIFY COLUMN phone VARCHAR(30) NOT NULL;

-- CHANGE: rename a column AND change its type
ALTER TABLE customers
  CHANGE COLUMN phone phone_number VARCHAR(30);

-- DROP a column
ALTER TABLE customers
  DROP COLUMN middle_name;

-- Multiple changes in one statement (more efficient)
ALTER TABLE customers
  ADD COLUMN    newsletter  BOOLEAN      DEFAULT FALSE,
  ADD COLUMN    country     VARCHAR(60)  DEFAULT 'India',
  MODIFY COLUMN email       VARCHAR(150) NOT NULL;
sql — DDL DROP vs TRUNCATE — know the difference 
-- TRUNCATE: removes all rows, keeps structure, resets AUTO_INCREMENT
-- Fast. Cannot be rolled back in MySQL (implicit commit).
TRUNCATE TABLE customers;

-- DROP TABLE: removes table + all data permanently
DROP TABLE customers;

-- Safe version: only drop if it exists
DROP TABLE IF EXISTS customers;

-- DROP DATABASE: deletes entire database and all its tables
DROP DATABASE myshop;
DROP DATABASE IF EXISTS myshop;

-- Comparison summary:
-- DELETE FROM table    → removes rows, can be WHERE filtered, rollback-able
-- TRUNCATE TABLE       → removes ALL rows, resets ID, NOT rollback-able
-- DROP TABLE           → removes the table itself entirely
💡Think of it this way: DELETE is like removing items from a box, TRUNCATE empties the box instantly, DROP destroys the box itself.
sql — DDL Rename tables and columns 
-- Rename a table (two equivalent ways)
RENAME TABLE customers TO users;

ALTER TABLE customers RENAME TO users;

-- Rename multiple tables at once
RENAME TABLE
  customers TO users,
  products  TO items;

-- Rename a column (must specify full type again)
ALTER TABLE users
  RENAME COLUMN customer_id TO user_id;  -- MySQL 8.0+

-- Older MySQL: use CHANGE COLUMN instead
ALTER TABLE users
  CHANGE COLUMN customer_id user_id INT AUTO_INCREMENT PRIMARY KEY;
FRI Basic SELECT — Querying Your Data DQL ▶

SELECT is the most used SQL statement — you'll write it hundreds of times a day. Master the fundamentals here and everything else builds on top. SQL clauses must be written in a specific order, and they are also executed in a specific internal order.

ℹ️Clause order (write order): SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT
Execution order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
sql — DQL SELECT fundamentals — use these to query customers 
-- Select ALL columns (avoid in production — fetches unnecessary data)
SELECT * FROM customers;

-- Select SPECIFIC columns (always prefer this)
SELECT first_name, last_name, email
FROM customers;

-- Select with a calculated column
SELECT
  first_name,
  last_name,
  price,
  price * 1.18  -- price with 18% GST
FROM products;

-- Literal values (no table needed)
SELECT 100 + 200;         -- returns 300
SELECT 'Hello', NOW();    -- returns string and current time
sql — DQL WHERE clause — filter rows by condition 
-- Basic comparison operators: = , != , < , > , <= , >=
SELECT * FROM products WHERE price > 500;
SELECT * FROM customers WHERE is_active = 1;

-- AND / OR / NOT
SELECT * FROM products
WHERE price > 100 AND stock_qty > 0;

SELECT * FROM customers
WHERE country = 'India' OR country = 'USA';

-- IN: match any value in a list
SELECT * FROM products
WHERE category IN ('electronics', 'books');

-- BETWEEN: inclusive range
SELECT * FROM products
WHERE price BETWEEN 200 AND 1000;

-- LIKE: pattern matching (% = any chars, _ = one char)
SELECT * FROM customers
WHERE email LIKE '%@gmail.com';

SELECT * FROM customers
WHERE first_name LIKE 'A%';       -- starts with A

SELECT * FROM customers
WHERE phone LIKE '98_______';    -- 10-digit starting 98

-- IS NULL / IS NOT NULL
SELECT * FROM customers WHERE phone IS NULL;
SELECT * FROM customers WHERE phone IS NOT NULL;
⚠️Never use = NULL to check for nulls — it always returns false. Always use IS NULL or IS NOT NULL.
sql — DQL ORDER BY, LIMIT, OFFSET 
-- ORDER BY: sort results (ASC = default, DESC = reverse)
SELECT * FROM products
ORDER BY price ASC;           -- cheapest first

SELECT * FROM products
ORDER BY price DESC;          -- most expensive first

-- Sort by multiple columns
SELECT * FROM customers
ORDER BY last_name ASC, first_name ASC;

-- LIMIT: max rows to return
SELECT * FROM products
ORDER BY price DESC
LIMIT 5;                       -- top 5 most expensive

-- LIMIT with OFFSET: pagination
-- Page 1: rows 1–10
SELECT * FROM products LIMIT 10 OFFSET 0;

-- Page 2: rows 11–20
SELECT * FROM products LIMIT 10 OFFSET 10;

-- Page 3: rows 21–30
SELECT * FROM products LIMIT 10 OFFSET 20;

-- Short form: LIMIT offset, count
SELECT * FROM products LIMIT 20, 10;  -- skip 20, get next 10
sql — DQL Column aliases and DISTINCT 
-- Column alias with AS (renames output column)
SELECT
  first_name  AS fname,
  last_name   AS lname,
  price * 1.18 AS price_with_gst
FROM products;

-- Table alias (shortens long table names)
SELECT p.name, p.price
FROM products p;    -- 'p' is the alias for products

-- DISTINCT: remove duplicate values from results
SELECT DISTINCT category FROM products;
-- Returns each unique category once

SELECT DISTINCT country, department
FROM employees;
-- Unique combinations of country + department

-- Count distinct values
SELECT COUNT(DISTINCT category) AS total_categories
FROM products;

Week 2 — DML: Data Manipulation
INSERT · UPDATE · DELETE · REPLACE · TRUNCATE
Week 2
DML INSERT — Adding Data to Tables Write ▶

INSERT statements add new rows to a table. You can insert a single row, multiple rows in one statement, or insert from a SELECT query.

💡Always list columns explicitly — it makes your code resilient to future schema changes.
sql — INSERT patterns
-- Single row insert
INSERT INTO users (name, email, age)
VALUES ('Alice', 'alice@example.com', 28);

-- Multiple rows in one shot (faster than individual inserts)
INSERT INTO products (name, price, stock)
VALUES
  ('Laptop',  75000.00, 10),
  ('Mouse',    1200.00, 50),
  ('Keyboard', 2500.00, 30);

-- Insert from a SELECT (copy rows between tables)
INSERT INTO archive_orders
SELECT * FROM orders
WHERE created_at < '2024-01-01';

-- INSERT IGNORE — skip rows that cause duplicate-key errors
INSERT IGNORE INTO users (email) VALUES ('bob@example.com');

-- ON DUPLICATE KEY UPDATE — upsert pattern
INSERT INTO product_views (product_id, views)
VALUES (42, 1)
ON DUPLICATE KEY UPDATE views = views + 1;
DML UPDATE & DELETE — Modifying and Removing Data Modify ▶

UPDATE changes existing rows. DELETE removes rows. Both MUST use a WHERE clause in production — without one, every row is affected.

⚠️Always test your WHERE clause with a SELECT first before running UPDATE or DELETE. Use transactions to protect yourself.
sql — UPDATE & DELETE
-- Update a single column for one row
UPDATE users
SET email = 'newemail@example.com'
WHERE user_id = 5;

-- Update multiple columns
UPDATE products
SET price = price * 1.10,
    updated_at = NOW()
WHERE category = 'electronics';

-- Safe delete pattern: test first
SELECT * FROM orders WHERE status = 'cancelled';
-- Then delete
DELETE FROM orders WHERE status = 'cancelled';

-- DELETE with LIMIT (useful for large batch deletions)
DELETE FROM logs
WHERE created_at < '2023-01-01'
LIMIT 1000;

-- TRUNCATE: wipe all rows, reset AUTO_INCREMENT (no rollback!)
TRUNCATE TABLE temp_data;

Week 3
JOIN Foreign Keys & Table Relationships Schema ▶

A Foreign Key (FK) links a column in one table to the PRIMARY KEY of another, enforcing referential integrity — you cannot insert a child row that has no parent.

sql — Foreign Keys
CREATE TABLE customers (
  customer_id INT AUTO_INCREMENT PRIMARY KEY,
  name        VARCHAR(100) NOT NULL,
  email       VARCHAR(150) UNIQUE NOT NULL
);

CREATE TABLE orders (
  order_id    INT AUTO_INCREMENT PRIMARY KEY,
  customer_id INT NOT NULL,
  total       DECIMAL(10,2),
  order_date  DATE,
  FOREIGN KEY (customer_id)
    REFERENCES customers(customer_id)
    ON DELETE CASCADE       -- deleting a customer removes their orders
    ON UPDATE CASCADE
);
JOIN All JOIN Types with Examples Query ▶

Joins combine rows from two or more tables based on a related column. Each join type returns a different subset of matching/non-matching rows.

ℹ️INNER JOIN = only matching rows. LEFT JOIN = all from left + matches from right (NULLs where no match). RIGHT JOIN = inverse of LEFT.
sql — JOIN types
-- INNER JOIN: only rows with a match in both tables
SELECT c.name, o.order_id, o.total
FROM customers c
INNER JOIN orders o ON c.customer_id = o.customer_id;

-- LEFT JOIN: all customers, even those with no orders (NULL)
SELECT c.name, o.order_id
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id;

-- Find customers with NO orders (anti-join)
SELECT c.name
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;

-- SELF JOIN: employees and their managers (same table)
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.employee_id;

-- Multi-table join
SELECT c.name, p.name AS product, oi.quantity
FROM customers c
JOIN orders o      ON c.customer_id = o.customer_id
JOIN order_items oi ON o.order_id    = oi.order_id
JOIN products p     ON oi.product_id = p.product_id;

Week 4
AGG GROUP BY, HAVING & Aggregate Functions Analytics ▶

Aggregate functions collapse many rows into a single value. GROUP BY groups rows that share a value; HAVING filters groups (like WHERE but for grouped results).

💡Rule: every column in SELECT that is not inside an aggregate function MUST appear in GROUP BY.
sql — Aggregate functions
-- Basic aggregates
SELECT
  COUNT(*),                         -- total rows
  COUNT(DISTINCT customer_id),      -- unique customers
  SUM(total),                        -- sum of totals
  AVG(total),                        -- average order value
  MIN(total), MAX(total)            -- min and max
FROM orders;

-- GROUP BY: sales by category
SELECT category, COUNT(*) AS items, SUM(price) AS total_value
FROM products
GROUP BY category;

-- HAVING: only categories with more than 5 products
SELECT category, COUNT(*) AS cnt
FROM products
GROUP BY category
HAVING COUNT(*) > 5
ORDER BY cnt DESC;

-- String functions
SELECT
  UPPER(name),
  LOWER(email),
  LENGTH(name),
  CONCAT(first_name, ' ', last_name) AS full_name,
  SUBSTRING(phone, 1, 3) AS area_code,
  TRIM(description)
FROM users;

-- Date functions
SELECT
  NOW(),
  CURDATE(),
  YEAR(created_at),
  MONTH(created_at),
  DATEDIFF(NOW(), created_at) AS days_old,
  DATE_FORMAT(created_at, '%d %M %Y') AS pretty_date
FROM orders;

Week 5 — Indexes & Performance
B-Tree Indexes · Composite · EXPLAIN · Query Optimization
Week 5
IDX Creating & Using Indexes Performance ▶

An index is a data structure (B-Tree by default) that MySQL builds to speed up lookups. Without indexes, MySQL scans every row (full table scan). With them, it jumps straight to the result.

⚠️Indexes speed up reads but slow down writes (INSERT/UPDATE/DELETE). Don't over-index — index columns you frequently filter, sort, or join on.
sql — Indexes
-- Create a simple index
CREATE INDEX idx_email ON users(email);

-- Unique index (enforces uniqueness + speeds lookups)
CREATE UNIQUE INDEX idx_username ON users(username);

-- Composite index (covers queries filtering on both columns)
CREATE INDEX idx_cat_price ON products(category, price);

-- List all indexes on a table
SHOW INDEX FROM users;

-- EXPLAIN: see how MySQL executes a query
EXPLAIN SELECT * FROM orders
WHERE customer_id = 42
  AND status = 'shipped';
-- Look at 'type' column: ref/range = good, ALL = full scan (needs index)

-- Drop an index
DROP INDEX idx_email ON users;

Week 6
TXN Transactions — ACID Guarantees Safety ▶

A transaction groups multiple SQL statements into a single all-or-nothing unit. ACID stands for Atomicity, Consistency, Isolation, Durability — the four properties that guarantee data integrity.

ℹ️Atomicity: all or nothing. Consistency: DB moves from valid state to valid state. Isolation: concurrent transactions don't interfere. Durability: committed data survives crashes.
sql — Transactions
-- Classic bank transfer example
START TRANSACTION;

UPDATE accounts
  SET balance = balance - 500
  WHERE account_id = 1;   -- debit sender

UPDATE accounts
  SET balance = balance + 500
  WHERE account_id = 2;   -- credit receiver

-- If everything is OK:
COMMIT;

-- If something went wrong, undo everything:
ROLLBACK;

-- SAVEPOINT: partial rollbacks
START TRANSACTION;
INSERT INTO orders (customer_id, total) VALUES (1, 200);
SAVEPOINT after_order;

INSERT INTO order_items (order_id, product_id) VALUES (LAST_INSERT_ID(), 99);
-- Item failed? Roll back only the item
ROLLBACK TO after_order;
COMMIT;

Week 7
CTE CTEs (WITH clause) & Subqueries Advanced SQL ▶

A CTE (Common Table Expression) is a named temporary result set defined with WITH. It makes complex queries readable and avoids repeating subquery logic. Recursive CTEs can traverse hierarchical data.

sql — CTEs & Subqueries
-- Basic CTE: top customers by spend
WITH customer_totals AS (
  SELECT customer_id, SUM(total) AS lifetime_value
  FROM orders
  GROUP BY customer_id
)
SELECT c.name, ct.lifetime_value
FROM customers c
JOIN customer_totals ct ON c.customer_id = ct.customer_id
ORDER BY ct.lifetime_value DESC
LIMIT 10;

-- Recursive CTE: walk an employee hierarchy
WITH RECURSIVE org_chart AS (
  -- Anchor: start with the CEO (no manager)
  SELECT employee_id, name, manager_id, 0 AS level
  FROM employees
  WHERE manager_id IS NULL
  UNION ALL
  -- Recursive: join to find each person's reports
  SELECT e.employee_id, e.name, e.manager_id, oc.level + 1
  FROM employees e
  JOIN org_chart oc ON e.manager_id = oc.employee_id
)
SELECT REPEAT('  ', level), name, level
FROM org_chart;

-- Correlated subquery: products priced above their category average
SELECT name, price, category
FROM products p
WHERE price > (
  SELECT AVG(price)
  FROM products
  WHERE category = p.category
);
VIEW Views & User Privileges Security ▶

A VIEW is a stored SELECT statement that acts like a virtual table. Use views to simplify complex queries, abstract schema changes, and control what data users can see.

sql — Views & Privileges
-- Create a view (hides sensitive columns)
CREATE VIEW public_users AS
  SELECT user_id, username, created_at
  FROM users
  WHERE is_active = TRUE;

-- Query the view just like a table
SELECT * FROM public_users;

-- Update a view
CREATE OR REPLACE VIEW public_users AS
  SELECT user_id, username, email, created_at
  FROM users WHERE is_active = TRUE;

-- User management & privileges
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'StrongPass!1';

-- Grant read-only access to one database
GRANT SELECT ON myapp.* TO 'app_user'@'localhost';

-- Grant write access to a specific table only
GRANT INSERT, UPDATE ON myapp.orders TO 'app_user'@'localhost';

FLUSH PRIVILEGES;
SHOW GRANTS FOR 'app_user'@'localhost';

WIN Window Functions — OVER, PARTITION BY, RANK Expert ▶

Window functions perform calculations across a set of rows related to the current row — without collapsing them like GROUP BY. They're the key to advanced analytics queries.

sql — Window Functions
-- ROW_NUMBER: rank orders within each customer
SELECT
  customer_id, order_id, total,
  ROW_NUMBER() OVER (
    PARTITION BY customer_id
    ORDER BY order_date DESC
  ) AS order_rank
FROM orders;

-- Get only the most recent order per customer
WITH ranked AS (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC) AS rn
  FROM orders
)
SELECT * FROM ranked WHERE rn = 1;

-- Running total (cumulative SUM)
SELECT
  order_date, total,
  SUM(total) OVER (ORDER BY order_date) AS running_total
FROM orders;

-- RANK vs DENSE_RANK
SELECT name, score,
  RANK()       OVER (ORDER BY score DESC) AS rank_with_gaps,
  DENSE_RANK() OVER (ORDER BY score DESC) AS dense_rank
FROM leaderboard;
PROC Stored Procedures, Functions & Triggers Expert ▶

Stored procedures are reusable SQL programs stored in the database. Triggers fire automatically on INSERT/UPDATE/DELETE events. Both keep business logic inside the DB layer.

sql — Stored Procedure & Trigger
DELIMITER //

-- Stored Procedure: place an order with stock check
CREATE PROCEDURE PlaceOrder(
  IN  p_customer_id INT,
  IN  p_product_id  INT,
  IN  p_qty         INT,
  OUT p_order_id    INT
)
BEGIN
  DECLARE v_stock INT;
  SELECT stock INTO v_stock FROM products WHERE product_id = p_product_id;

  IF v_stock < p_qty THEN
    SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'Insufficient stock';
  END IF;

  START TRANSACTION;
    INSERT INTO orders (customer_id) VALUES (p_customer_id);
    SET p_order_id = LAST_INSERT_ID();
    UPDATE products SET stock = stock - p_qty WHERE product_id = p_product_id;
  COMMIT;
END//

-- Trigger: log price changes automatically
CREATE TRIGGER after_price_update
AFTER UPDATE ON products
FOR EACH ROW
BEGIN
  IF OLD.price != NEW.price THEN
    INSERT INTO price_audit (product_id, old_price, new_price, changed_at)
    VALUES (OLD.product_id, OLD.price, NEW.price, NOW());
  END IF;
END//

DELIMITER ;

-- Call the procedure
CALL PlaceOrder(1, 42, 2, @new_order_id);
SELECT @new_order_id;
JSON JSON Columns & Full-Text Search Expert ▶

MySQL 5.7+ supports a native JSON column type with functions to store, query, and manipulate JSON data. Useful for semi-structured or schema-flexible data.

sql — JSON & Full-Text
-- JSON column in a table
CREATE TABLE products (
  product_id INT PRIMARY KEY,
  name       VARCHAR(200),
  metadata   JSON
);

INSERT INTO products VALUES (
  1, 'Laptop',
  '{"color":"silver","ram_gb":16,"tags":["portable","work"]}'
);

-- Extract a JSON field (-> operator)
SELECT name, metadata->'$.color' AS color
FROM products;

-- Filter on JSON value
SELECT * FROM products
WHERE metadata->>'$.ram_gb' >= 16;

-- Full-text search
ALTER TABLE articles
  ADD FULLTEXT INDEX ft_content(title, body);

SELECT title, MATCH(title, body) AGAINST ('MySQL performance') AS relevance
FROM articles
WHERE MATCH(title, body) AGAINST ('MySQL performance' IN BOOLEAN MODE)
ORDER BY relevance DESC;
