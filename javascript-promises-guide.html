Why Promises?

The problem they solve — callback hell

JavaScript is single-threaded and non-blocking. Long-running operations (network requests, file I/O, timers) are handled asynchronously. Traditionally, this meant callbacks — functions passed as arguments to be called later.

The Callback Problem

Callbacks work fine for simple cases. But real applications have sequential async operations — and callbacks nest deeply:

// Callback Hell — the "Pyramid of Doom"
getUser(userId, function(err, user) {
  if (err) return handleError(err);

  getOrders(user.id, function(err, orders) {
    if (err) return handleError(err);

    getProduct(orders[0].productId, function(err, product) {
      if (err) return handleError(err);

      getReviews(product.id, function(err, reviews) {
        if (err) return handleError(err);
        // Finally use the data — 4 levels deep!
        console.log(reviews);
      });
    });
  });
});
Problems with Callbacks

Inversion of control (you hand off your code to a third party), hard error propagation, impossible to parallelize cleanly, untestable, and deeply nested — visually and logically confusing.

Promises solve all of this. They represent the eventual result of an async operation — an object you can attach handlers to, chain, compose, and reason about like any other value.

// 02

What is a Promise?

A proxy for a future value

A Promise is an object representing the eventual completion or failure of an asynchronous operation. Think of it like a restaurant buzzer — you get it immediately, and it will eventually notify you when your table (or value) is ready.

Formally: a Promise is a thenable object with a .then() method that follows the Promises/A+ spec.

Creating a Promise

const myPromise = new Promise(function(resolve, reject) {
  // executor function — runs synchronously, immediately

  const success = true;

  if (success) {
    resolve('It worked!');   // fulfills the promise
  } else {
    reject(new Error('Something failed'));  // rejects the promise
  }
});

JS
Key Facts

The executor function runs synchronously when the Promise is created. Only resolve and reject are asynchronous — their callbacks are scheduled as microtasks. Calling resolve or reject more than once has no effect — only the first call counts.

Arrow Function Syntax (Modern)

const fetchData = new Promise((resolve, reject) => {
  setTimeout(() => {
    resolve({ id: 1, name: 'Alice' });
  }, 1000);
});

pending The initial state. The operation hasn't completed yet. No
fulfilled The operation completed successfully. Has a result value. Yes

Immutability

Once a Promise transitions from pending to either fulfilled or rejected, it is settled and its state can never change again. This immutability is fundamental — it makes Promises safe to pass around and attach multiple handlers to.

Anatomy of a Promise

The executor, resolve, and reject — in detail

const promise = new Promise((resolve, reject) => {
  // ┌──────────────────────────────────────────┐
  // │  EXECUTOR FUNCTION                       │
  // │  Runs SYNCHRONOUSLY when Promise created │
  // └──────────────────────────────────────────┘

  // resolve(value) → fulfills the Promise
  // reject(reason) → rejects the Promise

  // Only the FIRST call to resolve/reject matters
  resolve(42);
  resolve(99);  // ← ignored, already settled
  reject('error');  // ← also ignored
});

// Throwing inside the executor rejects the Promise
const badPromise = new Promise((resolve, reject) => {
  throw new Error('Oops!');  // ← auto-rejected
});

resolve() nuances

When you call resolve(value), if value is itself a Promise (or thenable), the outer promise follows the inner one. It doesn't wrap it — it adopts its state.

const inner = Promise.resolve(42);
const outer = new Promise(resolve => resolve(inner));

// outer doesn't become Promise<Promise<42>>
// outer becomes Promise<42> — it unwraps!
outer.then(v => console.log(v)); // → 42
// 05

.then()

Attaching fulfillment and rejection handlers

.then(onFulfilled, onRejected) is the core method of every Promise. It takes up to two callback functions and always returns a new Promise.

promise.then(
  (value) => {
    // called if promise FULFILLS
    // value = the resolved value
    console.log('Success:', value);
    return 'next value';  // passed to next .then()
  },
  (reason) => {
    // called if promise REJECTS
    // reason = the rejection reason
    console.log('Error:', reason);
  }
);

What .then() returns

The new Promise returned by .then() resolves based on what the callback does:

const p = Promise.resolve(10);

p.then(v => v * 2)           // returns Promise<20>
 .then(v => v + 5)           // returns Promise<25>
 .then(v => console.log(v)); // logs 25

// Throwing in .then() rejects the returned Promise
p.then(v => {
  throw new Error('Nope!');
}).catch(err => console.log(err.message)); // → 'Nope!'

// Returning a Promise from .then() — it's flattened!
p.then(v => Promise.resolve(v * 3))  // NOT Promise<Promise<30>>
 .then(v => console.log(v));          // → 30 ✓
Handlers are always async

Even if a Promise is already fulfilled, .then() callbacks never run synchronously. They are always scheduled as microtasks, ensuring consistent behaviour.

// 06

.catch()

Handling rejections elegantly

.catch(onRejected) is syntactic sugar for .then(undefined, onRejected). It's the preferred way to handle errors in Promise chains.

Promise.reject(new Error('Something broke'))
  .catch(err => {
    console.log(err.message); // 'Something broke'
    // If you return here, the chain CONTINUES as fulfilled
    return 'recovered';
  })
  .then(v => console.log(v)); // → 'recovered'

Catch catches everything upstream

fetch('/api/data')
  .then(res => res.json())
  .then(data => processData(data))
  .then(result => displayResult(result))
  .catch(err => {
    // Catches rejections from ANY of the above .then() steps
    // Including thrown errors inside handlers
    console.error('Pipeline failed:', err);
  });
Always add a .catch()

Unhandled Promise rejections crash Node.js (since v15) and show warnings in browsers. Always terminate Promise chains with a .catch(), or use async/await with try/catch.

// 07

.finally()

Cleanup regardless of outcome

.finally(onFinally) (ES2018) runs its callback when the Promise settles — whether fulfilled or rejected. It does not receive the value or reason.

let loading = true;

fetchData()
  .then(data => render(data))
  .catch(err => showError(err))
  .finally(() => {
    loading = false;   // Always runs — perfect for cleanup
    hideSpinner();
  });

Key behaviours of .finally()

  • Passes through the value/reason transparently (doesn't change them)
  • If the callback throws or returns a rejected Promise, that rejection propagates
  • Returns a new Promise (like all Promise methods)
Promise.resolve(42)
  .finally(() => console.log('cleanup'))
  .then(v => console.log(v));
// logs: 'cleanup', then 42 — value passed through

Promise Chaining

Composing sequential async operations

Because .then() always returns a new Promise, you can chain calls to express sequential async operations in a flat, readable way — the antidote to callback hell.

fetch('https://api.example.com/users/1')
  .then(response => {
    if (!response.ok) throw new Error(`HTTP error: ${response.status}`);
    return response.json();             // returns a Promise!
  })
  .then(user => {
    console.log('Got user:', user.name);
    return fetch(`/api/orders/${user.id}`);  // another Promise
  })
  .then(response => response.json())
  .then(orders => {
    console.log('Orders:', orders);
    return orders;
  })
  .catch(err => console.error('Failed:', err));
Common mistake: not returning

If you forget to return a Promise inside a .then() handler, the chain doesn't wait for it. The next .then() runs immediately with undefined. Always return Promises from handlers!

// ❌ BUG — missing return
fetchUser()
  .then(user => {
    fetchOrders(user.id);  // not returned!
  })
  .then(orders => console.log(orders)); // → undefined

// ✅ CORRECT
fetchUser()
  .then(user => return fetchOrders(user.id))
  .then(orders => console.log(orders)); // ← works!

Promise.resolve() & Promise.reject()

Creating instantly settled Promises

// Promise.resolve(value) — creates an already-fulfilled Promise
const p1 = Promise.resolve(42);
p1.then(v => console.log(v)); // → 42

// Passing a thenable — it's unwrapped
const p2 = Promise.resolve(Promise.resolve(99));
p2.then(v => console.log(v)); // → 99 (not Promise<99>)

// Promise.reject(reason) — creates an already-rejected Promise
const p3 = Promise.reject(new Error('Nope'));
p3.catch(err => console.log(err.message)); // → 'Nope'

// Common use: normalizing sync values in async functions
function maybeAsync(value) {
  if (value > 0) return Promise.resolve(value);
  return Promise.reject(new Error('Value must be positive'));
}

Promise.all()

Wait for all — fail fast on any rejection

Promise.all(iterable) takes an iterable of Promises and returns a new Promise that:

  • Fulfills when ALL promises fulfill — with an array of their values (in order)
  • Rejects immediately when ANY promise rejects — with that reason
const p1 = fetch('/api/users').then(r => r.json());
const p2 = fetch('/api/products').then(r => r.json());
const p3 = fetch('/api/orders').then(r => r.json());

Promise.all([p1, p2, p3])
  .then(([users, products, orders]) => {
    // All 3 completed! Results in same order as input
    console.log(users, products, orders);
  })
  .catch(err => {
    // If ANY one fails, we're here immediately
    console.error('One request failed:', err);
  });
All-or-nothing semantics

Promise.all() is "fail-fast" — if one rejects, you lose all results. The other Promises still run to completion, but their values are discarded. If you need all results regardless, use Promise.allSettled().

Non-Promise values are allowed

Promise.all([1, Promise.resolve(2), 'three'])
  .then(values => console.log(values)); // → [1, 2, 'three']

Promise.allSettled()

Wait for all, regardless of failure — ES2020

Promise.allSettled(iterable) always fulfills (never rejects) when all promises settle. Each result is a descriptor object:

const promises = [
  Promise.resolve('success'),
  Promise.reject(new Error('failed')),
  Promise.resolve(42),
];

Promise.allSettled(promises).then(results => {
  results.forEach(result => {
    if (result.status === 'fulfilled') {
      console.log('Value:', result.value);
    } else {
      console.log('Reason:', result.reason.message);
    }
  });
});
// Value: 'success'
// Reason: 'failed'
// Value: 42
Result statusProperties
fulfilled{ status: 'fulfilled', value: ... }
rejected{ status: 'rejected', reason: ... }

Promise.any()

First to fulfill wins — ES2021

Promise.any(iterable) returns a Promise that:

  • Fulfills with the value of the first fulfilled Promise
  • Rejects with an AggregateError only if all promises reject
const mirrors = [
  fetch('https://cdn1.example.com/file'),
  fetch('https://cdn2.example.com/file'),
  fetch('https://cdn3.example.com/file'),
];

// Use whichever CDN responds first
Promise.any(mirrors)
  .then(response => response.blob())
  .catch(err => {
    // AggregateError if ALL three fail
    console.log(err.errors); // array of all rejection reasons
  });
AggregateError

When all promises reject, Promise.any() rejects with an AggregateError which has an .errors property — an array of all individual rejection reasons, in input order.

// 13

Promise.race()

First to settle (fulfill OR reject) wins

Promise.race(iterable) returns a Promise that settles with the same value/reason as the first promise that settles — regardless of whether it fulfills or rejects.

// Classic: adding a timeout to any async operation
function withTimeout(promise, ms) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error(`Timed out after ${ms}ms`)), ms)
  );
  return Promise.race([promise, timeout]);
}

withTimeout(fetch('/api/slow-endpoint'), 5000)
  .then(res => res.json())
  .catch(err => console.error(err.message));
// → Either the data, or 'Timed out after 5000ms'
Promise.all()

Fulfills when ALL fulfill. Rejects immediately if ANY rejects. Returns array of values.

Promise.allSettled()

Always fulfills when ALL settle. Returns array of result descriptors. Never rejects.

Promise.any()

Fulfills when FIRST fulfills. Rejects with AggregateError if ALL reject.

Promise.race()

Settles (either way) with the FIRST to settle. Useful for timeouts.

async / await

Syntactic sugar over Promises — ES2017

async/await doesn't replace Promises — it is Promises, with cleaner syntax. An async function always returns a Promise. await pauses execution until the Promise settles.

// Promise chain approach
function getUser(id) {
  return fetch(`/api/users/${id}`)
    .then(res => res.json())
    .then(user => {
      return fetch(`/api/orders/${user.id}`)
        .then(res => res.json())
        .then(orders => ({ user, orders }));
    });
}

// Same thing with async/await — much cleaner!
async function getUser(id) {
  const userRes = await fetch(`/api/users/${id}`);
  const user = await userRes.json();

  const ordersRes = await fetch(`/api/orders/${user.id}`);
  const orders = await ordersRes.json();

  return { user, orders };  // async function wraps in Promise
}

await with Promise.all() for parallelism

async function loadDashboard() {
  // ❌ Sequential — 3 seconds total if each takes 1s
  const users = await fetchUsers();
  const stats = await fetchStats();
  const logs  = await fetchLogs();

  // ✅ Parallel — ~1 second total
  const [users, stats, logs] = await Promise.all([
    fetchUsers(),
    fetchStats(),
    fetchLogs(),
  ]);

  return { users, stats, logs };
}

async functions always return a Promise

async function add(a, b) {
  return a + b;  // wrapped in Promise.resolve()
}

add(1, 2).then(console.log); // → 3

// Throwing inside async rejects the returned Promise
async function fail() {
  throw new Error('async error');
}
fail().catch(err => console.log(err.message)); // → 'async error'

Error Handling

Catching, recovering, and propagating errors

try/catch with async/await

async function loadData() {
  try {
    const res = await fetch('/api/data');
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return await res.json();
  } catch (err) {
    if (err.name === 'TypeError') {
      console.error('Network error or CORS issue');
    } else {
      console.error('Request failed:', err.message);
    }
    throw err; // re-throw if you can't recover
  } finally {
    hideSpinner();
  }
}

Global unhandled rejection handler

// Browser
window.addEventListener('unhandledrejection', event => {
  console.error('Unhandled rejection:', event.reason);
  event.preventDefault(); // prevent default browser logging
});

// Node.js
process.on('unhandledRejection', (reason, promise) => {
  console.error('Unhandled rejection at:', promise, 'reason:', reason);
  process.exit(1); // exit with failure code
});
Error swallowing

An empty .catch() silently swallows errors. This is almost always a bug. At minimum, log the error. Only suppress errors when you've intentionally handled the failure case and the rest of the program can proceed normally.

Useful Patterns

Real-world recipes with Promises

Sequential Iteration

// Process items one-by-one (not in parallel)
async function processSequentially(items) {
  const results = [];
  for (const item of items) {
    const result = await processItem(item); // waits each time
    results.push(result);
  }
  return results;
}

// With reduce (without async/await)
function processSequentially(items) {
  return items.reduce(
    (chain, item) => chain.then(() => processItem(item)),
    Promise.resolve()
  );
}

Retry Logic

async function retry(fn, retries = 3, delay = 1000) {
  try {
    return await fn();
  } catch (err) {
    if (retries === 0) throw err;
    await new Promise(resolve => setTimeout(resolve, delay));
    return retry(fn, retries - 1, delay * 2); // exponential backoff
  }
}

const data = await retry(() => fetch('/api/flaky'), 3, 500);

Promisifying Callbacks

// Wrap any Node.js-style callback API
function promisify(fn) {
  return function(...args) {
    return new Promise((resolve, reject) => {
      fn(...args, (err, result) => {
        if (err) reject(err);
        else resolve(result);
      });
    });
  };
}

const readFile = promisify(fs.readFile);
const content = await readFile('./file.txt', 'utf8');
// Node.js also has util.promisify() built-in

Deferred Promise

// Expose resolve/reject outside the executor (use sparingly)
function createDeferred() {
  let resolve, reject;
  const promise = new Promise((res, rej) => {
    resolve = res;
    reject = rej;
  });
  return { promise, resolve, reject };
}

const { promise, resolve } = createDeferred();
resolve('done later!');
await promise; // → 'done later!'

Common Gotchas

Mistakes everyone makes — and how to avoid them

1. Forgetting to return

// ❌ WRONG
getData().then(data => {
  processData(data); // not returned — chain doesn't wait
}).then(result => console.log(result)); // undefined!

// ✅ RIGHT
getData().then(data => processData(data))
         .then(result => console.log(result));

2. await inside forEach

// ❌ WRONG — forEach doesn't await Promises!
async function badLoop(ids) {
  ids.forEach(async id => {
    await processItem(id); // fire and forget!
  });
  // Returns before any item is processed
}

// ✅ RIGHT — use for...of
async function goodLoop(ids) {
  for (const id of ids) {
    await processItem(id);
  }
}

3. Creating unnecessary Promises

// ❌ Promise constructor anti-pattern
function fetchData() {
  return new Promise((resolve, reject) => {
    fetch('/api/data')
      .then(res => res.json())
      .then(resolve)  // unnecessary wrapping
      .catch(reject);
  });
}

// ✅ SIMPLE — fetch already returns a Promise
function fetchData() {
  return fetch('/api/data').then(res => res.json());
}

4. Mixing async/await with .then()

// ❌ Confusing — pick one style per function
async function mixed() {
  return await fetchData()
    .then(data => process(data));
}

// ✅ Pure async/await
async function clean() {
  const data = await fetchData();
  return process(data);
}

The Microtask Queue

Why Promises are always asynchronous — even when settled

JavaScript has an event loop with multiple queues. Promise callbacks run in the microtask queue, which has higher priority than the macrotask queue (setTimeout, setInterval, I/O).

console.log('1 — synchronous');

setTimeout(() => console.log('2 — macrotask (setTimeout)'), 0);

Promise.resolve().then(() => console.log('3 — microtask'));

console.log('4 — synchronous');

// OUTPUT ORDER:
// 1 — synchronous
// 4 — synchronous
// 3 — microtask       ← before setTimeout!
// 2 — macrotask
Why this matters

Microtasks (Promises, queueMicrotask, MutationObserver) drain completely before the browser renders or the next macrotask starts. A long chain of Promises won't block rendering, but a tight loop of queueMicrotask calls can starve the event loop.

Event Loop Order

// 1. Execute current synchronous code (call stack)
// 2. Drain the ENTIRE microtask queue
// 3. Render (browser only)
// 4. Pick one macrotask (setTimeout, setInterval, etc.)
// 5. Go to step 2
