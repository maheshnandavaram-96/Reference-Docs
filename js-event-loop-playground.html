JS RUNTIME COMPONENTS

Heap

The Memory Heap is where JavaScript stores objects and variables in an unstructured region of memory. When you create an object like {} or an array, it lives here. The garbage collector automatically reclaims memory no longer referenced.

Call Stack

A LIFO (Last In, First Out) data structure that tracks the currently executing functions. When a function is called, a new frame is pushed onto the stack. When it returns, the frame is popped. JavaScript is single-threaded — only one function executes at a time via this stack. If the stack overflows (infinite recursion), you get a "Maximum call stack size exceeded" error.

Web APIs

APIs provided by the browser runtime (not the JS engine itself). These include setTimeout, fetch, XMLHttpRequest, localStorage, DOM events, and more. When you call setTimeout, the browser takes the callback and timer — freeing the Call Stack immediately. When the timer expires, the callback is moved to the Task Queue.

Event Loop

The heart of async JavaScript. The Event Loop continuously monitors two things: is the Call Stack empty? And are there pending callbacks? Its algorithm: (1) Run all synchronous code until stack is empty. (2) Drain the entire Microtask Queue. (3) Pick ONE task from the Task Queue. (4) Repeat. This loop is what makes JavaScript feel "concurrent" despite being single-threaded.

Task Queue (Macrotask Queue)

A FIFO queue for callbacks from Web APIs — primarily setTimeout, setInterval, setImmediate, and I/O events. The Event Loop picks exactly ONE task per loop iteration. Even setTimeout(fn, 0) queues here, so it never runs before the current synchronous code finishes.

Microtask Queue

A higher-priority queue for microtasks — primarily Promise.then(), Promise.catch(), queueMicrotask(), and the continuation after await. After every task (and after the initial script), the Event Loop drains the entire Microtask Queue before moving on. Microtasks can even schedule more microtasks — they all run before the next Task.
